# Day 37: Patterns Homework

## Что добавлено

- `Singleton`: ручной `AnalyticsService`, который создаётся один раз и используется в `NotificationService`, `AuthViewModel`, `TasksViewModel`.
- `Factory`: `StatusWidgetFactory` для `loading/success/error` виджетов по `enum StatusWidgetType`.
- `Factory` в реальной задаче: `ApiResponseParserFactory` для парсинга удалённых ответов в `TaskModel`, `CommentModel`, `NotificationPayload`.

## Сравнение с альтернативой: DI через `get_it`

1. Ручной `Singleton` проще показать и объяснить на уроке.
Минус: он сильнее связывает код с конкретной реализацией и глобальным состоянием.

2. `get_it` удобнее для подмены зависимостей на тестах и для масштабирования проекта.
Минус: для маленьких задач настройка контейнера иногда выглядит тяжелее, чем прямой `factory`/`singleton`.

3. Ручной `Singleton` гарантирует единый экземпляр без дополнительной конфигурации.
Минус: жизненный цикл сложнее контролировать, если нужен reset, lazy reload или разные окружения.

4. `get_it` делает зависимости явными в `service_locator.dart` и помогает централизовать создание объектов.
Минус: появляется скрытая магия резолва, и новичку иногда труднее понять, откуда взялся объект.

5. Factory-подход хорошо подходит для выбора поведения по `enum` или типу ответа.
Минус: если фабрик становится слишком много, код может расползтись, и тогда DI-контейнер лучше управляет общей композицией.
